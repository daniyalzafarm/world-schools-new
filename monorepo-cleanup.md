# Monorepo Cleanup & Cursor Performance Guide

This guide is intended for developers working on this Nx-based monorepo with Next.js frontend apps and NestJS backend apps.

It explains how to keep the repo lightweight, avoid huge local caches, prevent Cursor/Claude from indexing or reading unnecessary folders, and safely clean generated files.

---

## Why This Matters

In large Nx/Next.js/NestJS monorepos, generated folders can grow very quickly. In one cleanup, the local repo had grown to more than 100GB mainly because of the Nx cache:

```text
.nx/    ~100GB
```

These folders are generated and should not be committed. They can also slow down Cursor/VS Code because the editor may watch, index, or scan them.

---

## Common Generated Folders

| Folder          | Purpose                      | Safe to delete? | Regenerated by            |
| --------------- | ---------------------------- | --------------: | ------------------------- |
| `node_modules/` | Installed npm dependencies   |             Yes | `npm ci` / `npm install`  |
| `.nx/`          | Nx cache and workspace data  |             Yes | Nx commands               |
| `.next/`        | Next.js build/dev cache      |             Yes | `next dev` / `next build` |
| `.turbo/`       | Turborepo cache, if used     |             Yes | Turbo commands            |
| `.cache/`       | General tool cache           |         Usually | Tool-specific             |
| `dist/`         | Build output                 |             Yes | Nx/TypeScript builds      |
| `build/`        | Build output                 |         Usually | Build scripts             |
| `coverage/`     | Test coverage output         |             Yes | Test commands             |
| `out/`          | Static export/build output   |         Usually | Next/static build         |
| `tmp/`          | Temporary files              |         Usually | Local tooling             |
| `*.tsbuildinfo` | TypeScript incremental cache |             Yes | TypeScript                |

## Safe Cleanup Commands

### Preview Cleanup Targets

Run this from the repo root first:

```bash
find . -type d \( \
  -name node_modules \
  -o -name .next \
  -o -name .nx \
  -o -name .turbo \
  -o -name .cache \
  -o -name dist \
  -o -name build \
  -o -name coverage \
  -o -name out \
\) -prune -exec du -sh {} \; 2>/dev/null | sort -h | tail -100
```

### Delete Generated Folders

Run from the repo root:

```bash
find . -type d \( \
  -name node_modules \
  -o -name .next \
  -o -name .nx \
  -o -name .turbo \
  -o -name .cache \
  -o -name dist \
  -o -name build \
  -o -name coverage \
  -o -name out \
\) -prune -exec rm -rf {} +
```

This removes generated folders and installed dependencies. After this, reinstall dependencies:

```bash
npm ci
```

or:

```bash
npm install
```

## Faster Cleanup Without Removing `node_modules`

If you want to keep dependencies installed and only remove build/cache folders:

```bash
find . -type d -name node_modules -prune -o -type d \( \
  -name .next \
  -o -name .nx \
  -o -name .turbo \
  -o -name .cache \
  -o -name dist \
  -o -name build \
  -o -name coverage \
  -o -name out \
\) -prune -exec rm -rf {} +
```

This is usually enough when `.nx/` or `.next/` grows too large.

> **Important:** The leading `-name node_modules -prune -o` is required. Without it, `find` descends **into** `node_modules` and matches the `dist/`, `build/`, `out/`, and `.cache/` folders that many npm packages ship as their actual published code — `rm -rf` then deletes them and corrupts the installed dependencies. Pruning `node_modules` first keeps it intact while still cleaning every build/cache folder outside it.

## App-Specific Cleanup

If only one app is large, inspect it:

```bash
du -sh apps/* 2>/dev/null | sort -h | tail -30
```

For example:

```bash
du -sh apps/wc-provider/* apps/wc-provider/.* 2>/dev/null | sort -h | tail -40
```

Find app-level generated folders:

```bash
find apps -type d \( \
  -name .next \
  -o -name .nx \
  -o -name .turbo \
  -o -name .cache \
  -o -name dist \
  -o -name build \
  -o -name coverage \
  -o -name out \
\) -prune -exec du -sh {} \; 2>/dev/null | sort -h | tail -50
```

Delete app-level generated folders:

```bash
find apps -type d -name node_modules -prune -o -type d \( \
  -name .next \
  -o -name .nx \
  -o -name .turbo \
  -o -name .cache \
  -o -name dist \
  -o -name build \
  -o -name coverage \
  -o -name out \
\) -prune -exec rm -rf {} +
```

## npm Cache

Check npm cache:

```bash
du -sh ~/.npm
```

Verify and garbage collect:

```bash
npm cache verify
```

Force clean:

```bash
npm cache clean --force
```

This does not change the repo, `package.json`, `package-lock.json`, or `node_modules`. It only clears npm’s package cache.

## `npm ci` vs `npm install`

### `npm ci`

Use for clean, reproducible installs.

Behavior:

1. Deletes existing `node_modules`
2. Installs exactly from `package-lock.json`

Good for:

- CI/CD
- Fresh clone
- Clean local reset
- When dependencies are behaving strangely

```bash
npm ci
```

### `npm install`

Use for regular local development.

Behavior:

- Keeps existing `node_modules`
- Adds/updates/removes packages as needed
- May update `package-lock.json`

```bash
npm install
```

## Cursor Performance Tips for This Monorepo

### 1. Open Only the Needed Folder When Possible

Instead of always opening the full monorepo, open only the app you are working on if practical:

```bash
cursor apps/wc-booking
cursor apps/wc-provider
cursor apps/wc-superadmin
cursor apps/wc-nest-api
```

### 2. Resync Cursor Index After Cleanup

After updating `.cursorignore` or deleting caches:

```text
Cursor Settings → Indexing → Resync Index
```

### 3. Disable Unneeded Extensions for This Workspace

Useful candidates to disable if not needed:

- Angular Language Service
- Claude Code extension
- Nx Console, if it is causing extra language-server processes
- Heavy database/Docker/Kubernetes extensions

Disable per workspace rather than globally when possible.

### 4. Debug Cursor Memory

Use RSS to inspect real memory usage:

```bash
ps -axo pid,rss,pmem,comm,args | grep -i cursor | sort -nrk 2 | head -30
```

If Claude Code is running and consuming resources:

```bash
pkill -f "anthropic.claude-code"
```

If Angular language service is running in a non-Angular project:

```bash
pkill -f "angular.ng-template"
```

## Recommended Cleanup Workflow

When the repo gets too large:

```bash
# 1. Check repo size
cd /path/to/repo
du -sh .

# 2. Preview generated folders
find . -type d \( \
  -name node_modules \
  -o -name .next \
  -o -name .nx \
  -o -name .turbo \
  -o -name .cache \
  -o -name dist \
  -o -name build \
  -o -name coverage \
  -o -name out \
\) -prune -exec du -sh {} \; 2>/dev/null | sort -h | tail -100

# 3. Delete generated folders
find . -type d \( \
  -name node_modules \
  -o -name .next \
  -o -name .nx \
  -o -name .turbo \
  -o -name .cache \
  -o -name dist \
  -o -name build \
  -o -name coverage \
  -o -name out \
\) -prune -exec rm -rf {} +

# 4. Reinstall dependencies
npm ci

# 5. Recheck size
du -sh .
```

## If You Want to Delete and Reclone the Repo

This is safe only if there is no uncommitted local work or local-only config.

Before deleting:

```bash
git status
git status --short
find . -maxdepth 3 -name "*.env" -o -name ".env*"
ls -la .claude .cursor .vscode 2>/dev/null
```

Safer approach: rename first:

```bash
cd /path/to/parent-folder
mv repo-name repo-name-backup
git clone <repo-url> repo-name
```

After confirming the new clone works:

```bash
rm -rf repo-name-backup
```

In most cases, deleting `.nx/`, `.next/`, build outputs, and `node_modules/` gives the same benefit faster than recloning.

## Contributor Notes

- Do not commit generated folders or caches.
- Keep `.gitignore`, `.cursorignore`, and `.claudeignore` updated when adding new tools.
- If an app creates local uploads, database files, browser sessions, or generated reports, add those paths to `.gitignore`.
- If Cursor becomes slow or memory-heavy, check indexing and extension processes before blaming the repo itself.
- When in doubt, preview with `du -sh` before deleting.
